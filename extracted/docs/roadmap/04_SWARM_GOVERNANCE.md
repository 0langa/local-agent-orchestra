# 04 — SWARM GOVERNANCE
## Central Controller Architecture, Swarm-Safe Execution, and Anti-Chaos Protocols

**Status:** DERIVED FROM 00_PROJECT_DOCTRINE
**Enforcement:** All swarm execution must follow these protocols.
**Violation Classification:** ARCHITECTURAL BREACH (Level 3)

## 1. Central Controller Architecture

### 1.1 Pattern: Trainable Orchestrator + Frozen Workers
The swarm follows a hierarchical task decomposition pattern:
- **Central Orchestrator:** Single trainable component that performs semantic parsing and dependency analysis
- **Task Dependency Graph (DAG):** Generated by the orchestrator to identify parallelizable components, serial constraints, and required specializations
- **Frozen Workers:** Parameter-fixed sub-agents spun up on demand for specific subtasks

### 1.2 Orchestrator Responsibilities
1. Parse the high-level objective into decomposable sub-problems
2. Generate the execution DAG with dependency edges
3. Identify which subtasks can execute in parallel
4. Assign appropriate agent specializations to each subtask
5. Monitor execution progress and detect failures
6. Handle replanning when subtasks fail or budgets are exceeded
7. Synthesize results from completed subtasks

### 1.3 Worker Agent Rules
1. Workers receive focused task descriptions with appropriate tools
2. Workers have independent execution budgets
3. Workers are stateless with respect to other workers
4. Workers report results back to the orchestrator
5. Workers do not communicate directly with other workers
6. Workers are terminated when their subtask completes or fails

### 1.4 Scale Parameters

| Parameter | K2.5 Baseline | K2.6 Target |
|-----------|---------------|-------------|
| Maximum sub-agents | 100 | 300 |
| Coordinated steps | 1,500 | 4,000 |
| Effective step budget | 150,000 | 1,200,000 |
| Long-horizon stability | Hours | 12+ hours |

## 2. Swarm Decomposition Strategy

### 2.1 Decomposition Principles
1. **Wide over deep:** Swarms excel at parallel search across many items. Deep sequential chains should be kept within single agents.
2. **Independent subtasks:** Subtasks must be genuinely independent. False parallelism causes coordination overhead that exceeds benefits.
3. **Bounded replanning:** The orchestrator must not enter infinite replanning loops. Each replan consumes budget.
4. **Clear success criteria:** Every subtask must have binary success criteria verifiable by the orchestrator.

### 2.2 Swarm-Aware Task Decomposition

Tasks suitable for swarm parallelization:
- Multiple independent file operations
- Batch analysis across unrelated components
- Parallel verification of multiple conditions
- Independent research queries
- Multi-file refactoring with no cross-dependencies

Tasks NOT suitable for swarm parallelization:
- Mathematical proofs requiring lemma chaining
- Complex debugging with sequential hypothesis testing
- Algorithm design requiring holistic optimization
- Creative writing needing consistent voice
- Any task where each step's output is the next step's required input

### 2.3 Serial Collapse Prevention
Serial collapse is the tendency to default to single-agent execution despite parallel capabilities.

**Detection:**
- Monitor average parallelism ratio (concurrent agents / total agents spawned)
- Target ratio > 0.5 for parallelizable workloads
- Alert when ratio drops below 0.3

**Prevention:**
- Explicit PARALLEL_SAFE markers in task specifications
- Minimum parallelism floor enforcement
- Review of sequential execution patterns in post-run analysis

## 3. Swarm-Safe Execution Protocols

### 3.1 Agent Lifecycle Protocol

```
SPAWN → ASSIGN → EXECUTE → REPORT → (VERIFY → ACCEPT|RETRY) → TERMINATE
```

| State | Allowed Transitions | Timeout |
|-------|-------------------|---------|
| SPAWN | ASSIGN | 30s |
| ASSIGN | EXECUTE, TERMINATE (if invalid) | 60s |
| EXECUTE | REPORT, TERMINATE (on fatal error) | Per-subtask budget |
| REPORT | VERIFY, TERMINATE | 30s |
| VERIFY | ACCEPT, RETRY, TERMINATE | 120s |
| RETRY | EXECUTE (bounded) | Per-subtask budget |
| ACCEPT | TERMINATE | 10s |
| TERMINATE | (final) | 10s |

### 3.2 Resource Contention Rules
1. No two agents may write to the same file simultaneously
2. Agents must acquire file locks before write operations
3. Read operations are lock-free but logged
4. Workspace subdirectories may be allocated exclusively to single agents
5. Shared memory access is mediated through the memory protocol

### 3.3 Failure Propagation Rules
1. A single worker failure does not fail the entire run
2. The orchestrator decides whether to retry, replan, or abort
3. Cascading failures trigger orchestrator-level error classification
4. Worker failures are logged with full context for debugging
5. Budget exhaustion in one worker does not consume the total swarm budget

### 3.4 Communication Protocol
- Workers communicate ONLY with the orchestrator
- No direct worker-to-worker communication
- All communication is logged in the ledger
- Messages are structured (JSON schema validated)
- Messages include sender, recipient, timestamp, and payload hash

## 4. Anti-Chaos Rules

### 4.1 Overlap Prevention
- Subsystem ownership is defined in 07_SUBSYSTEM_DEFINITIONS.md
- An agent may only modify files within its assigned subsystem
- Cross-subsystem changes require explicit integration gate approval
- Two agents may not be assigned the same file in the same phase

### 4.2 Architecture Drift Prevention
- All changes must preserve the Core Invariant (Law 1)
- Changes that introduce provider/workflow/tool specifics into core are rejected
- Architecture Decision Records (ADRs) must be updated for boundary changes
- Weekly architecture consistency checks are mandatory

### 4.3 Race Condition Prevention
- File-level locking for all write operations
- Atomic ledger appends (single writer per run)
- Workspace isolation for parallel agent execution
- Deterministic task ordering for dependent subtasks

### 4.4 Budget Exhaustion Prevention
- Per-agent step budgets enforced by the orchestrator
- Total swarm budget tracked centrally
- Early warning at 75%, 90%, and 95% budget consumption
- Graceful degradation: reduce parallelism as budget depletes

### 4.5 Deadlock Prevention
- File locks have timeouts (default 300s)
- Cycle detection in task dependency graph
- Maximum wait time for resource acquisition
- Forced termination and cleanup of hung agents

## 5. Governance Hierarchy for 300-Agent Scale

### 5.1 Organizational Tiers

```
Tier 1: Central Orchestrator (1 agent)
    |
    +-- Tier 2: Subsystem Leads (5-10 agents)
    |       |
    |       +-- Tier 3: Task Agents (20-50 agents)
    |               |
    |               +-- Tier 4: Worker Agents (50-240 agents)
    |
    +-- Tier 2: Integration Agents (3-5 agents)
    |       |
    |       +-- Tier 3: Validation Agents (10-20 agents)
    |
    +-- Tier 2: Governance Agents (2-3 agents)
            |
            +-- Tier 3: Monitoring Agents (5-10 agents)
```

### 5.2 Tier Responsibilities

**Tier 1 (Central Orchestrator):**
- Objective parsing and DAG generation
- Resource allocation and budget management
- Failure detection and replanning
- Final result synthesis

**Tier 2 (Subsystem Leads):**
- Oversee implementation within a subsystem boundary
- Validate subsystem-level integration
- Report progress and blockers to orchestrator
- Enforce subsystem-specific conventions

**Tier 3 (Task Agents):**
- Implement specific tasks within a subsystem
- Write tests for their implementations
- Validate their own work before reporting complete

**Tier 4 (Worker Agents):**
- Execute atomic implementation steps
- Generate code, tests, or documentation
- No decision-making authority

**Tier 2 (Integration Agents):**
- Cross-subsystem integration validation
- Interface compatibility checks
- End-to-end test execution

**Tier 2 (Governance Agents):**
- Monitor for architectural law violations
- Enforce coding standards and conventions
- Flag boundary violations for review

## 6. Claw Group Coordination (Heterogeneous Agents)

When coordinating heterogeneous agents (different models, different providers):
1. Capability requirements are declared in task specifications
2. The capability registry resolves requirements to configured models
3. Each agent group runs the model appropriate for its task class
4. Deep reasoning tasks → reasoning models
5. High-volume tasks → fast/cheap models
6. Domain-specific tasks → specialized models
